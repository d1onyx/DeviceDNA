# DeviceDNA handover and migration guide

Use this guide when the app must run under a new owner's Firebase, Cloudflare,
Neon, Google Play, and AdMob accounts.

The important rule is simple: the Android app talks only to the Cloudflare
Worker. The app never connects to Neon directly, and it never stores Neon
credentials.

## What Must Be Replaced

| Area | Value | Where it is configured | Required |
|---|---|---|---|
| Firebase Android config | `android/google-services.json` | Generated by `scripts/setup-firebase-auth.sh` | Yes |
| Firebase project id for Worker token checks | `FIREBASE_PROJECT_ID` | `backend/wrangler.toml` | Yes |
| Firebase Web API key for `/auth/google` and `/auth/refresh` | `FIREBASE_WEB_API_KEY` | Cloudflare secret and `backend/.dev.vars` | Yes |
| Neon Postgres connection string | `DATABASE_URL` | Cloudflare secret and `backend/.dev.vars` | Yes |
| Cloudflare Worker URL used by Android | `syncBaseUrl` | `local.properties` or `-PsyncBaseUrl=...` | Yes for release |
| Cloudflare KV namespace for Firebase public-key cache | `PUBLIC_JWK_CACHE_KV` id | `backend/wrangler.toml` | Yes |
| Google Play package name | `GOOGLE_PLAY_PACKAGE_NAME` | `backend/wrangler.toml` | Yes for real subscriptions |
| Google Play subscription product id | `GOOGLE_PLAY_PREMIUM_PRODUCT_ID` and `premiumSubProductId` | `backend/wrangler.toml` and `local.properties` | Yes for real subscriptions |
| Google Play service account | `GOOGLE_PLAY_SERVICE_ACCOUNT_EMAIL`, `GOOGLE_PLAY_PRIVATE_KEY` | Cloudflare secrets and `backend/.dev.vars` | Yes for real subscriptions |
| Internal manual subscription API key | `INTERNAL_API_KEY` | Cloudflare secret and `backend/.dev.vars` | Optional |
| Google Play RTDN webhook secret | `PLAY_RTDN_VERIFICATION_TOKEN` | Cloudflare secret + Google Cloud Pub/Sub push URL | Optional |
| AdMob ids | `adMobAppId`, `adMobBannerAdUnitId`, `adMobInterstitialAdUnitId` | `local.properties` or Gradle properties | Optional, but needed before release with real ads |

Do not mix Firebase projects. These three values must come from the same Firebase
project:

- `android/google-services.json` -> `project_info.project_id`
- `backend/wrangler.toml` -> `FIREBASE_PROJECT_ID`
- `FIREBASE_WEB_API_KEY`

If they do not match, authentication fails with `invalid_token` or `auth_failed`.

## Runtime Flow

```text
Android app
  -> Google Sign-In SDK gets a Google ID token
  -> POST /auth/google on Cloudflare Worker
  -> Worker calls Firebase Auth REST API
  -> Worker returns Firebase idToken + refreshToken
  -> Android calls /v1/* with Authorization: Bearer <firebase_id_token>
  -> Worker verifies Firebase token and live Firebase account
  -> Worker reads/writes Neon Postgres
```

The Worker stores:

- users in Neon `users`
- device snapshots in Neon `devices`
- Premium state in Neon `user_subscriptions`

## 1. Prepare Firebase

Prerequisites: Firebase CLI, Android SDK, Node 18+, Cloudflare account, Neon
account.

1. Log in to Firebase:

   ```bash
   firebase login
   ```

2. Create or choose the new owner's Firebase project.

3. Enable Google sign-in:

   ```text
   Firebase Console -> Authentication -> Sign-in method -> Google -> Enable
   ```

4. Get the Android debug SHA-1 for local testing:

   ```bash
   keytool -list -v -keystore ~/.android/debug.keystore -alias androiddebugkey \
     -storepass android -keypass android | grep SHA1
   ```

   For release builds, use the SHA-1 from the real release keystore or from Play
   App Signing.

5. Generate Firebase config files:

   ```bash
   ./scripts/setup-firebase-auth.sh <PROJECT_ID> <SHA1>
   ```

   This script registers Android package `com.devstdvad.devicedna`, adds the
   SHA-1, downloads `android/google-services.json`, optionally downloads the iOS
   plist, and rewrites `.firebaserc`.

6. Copy the Firebase Web API key from:

   ```text
   Firebase Console -> Project settings -> General -> Web API key
   ```

   You will use it as `FIREBASE_WEB_API_KEY` in the Cloudflare step.

## 2. Prepare Neon

1. Create a Neon project.
2. Copy the pooled Postgres connection string.
3. Apply committed Drizzle migrations:

   ```bash
   cd backend
   npm install
   DATABASE_URL="postgresql://USER:PASSWORD@HOST/db?sslmode=require" npm run db:migrate
   ```

Do not run `npm run db:generate` during handover unless the schema code was
changed. Generated SQL migrations are already committed in `backend/drizzle/`.

## 3. Prepare Cloudflare Worker

1. Edit `backend/wrangler.toml`.

   Replace:

   ```toml
   FIREBASE_PROJECT_ID = "YOUR_FIREBASE_PROJECT_ID"
   ```

   with the Firebase project id from `android/google-services.json`.

2. Optionally change the Worker name:

   ```toml
   name = "devicedna-sync"
   ```

   The deployed URL will look like:

   ```text
   https://<name>.<cloudflare-subdomain>.workers.dev
   ```

3. Keep the KV placeholder until deploy:

   ```toml
   id = "REPLACE_AFTER_wrangler_kv_namespace_create"
   ```

   `setup-cloudflare.sh` creates the KV namespace and writes the real id into
   `wrangler.toml`.

4. Deploy:

   ```bash
   cd backend
   npx wrangler login

   DATABASE_URL="postgresql://USER:PASSWORD@HOST/db?sslmode=require" \
   FIREBASE_WEB_API_KEY="AIzaSy..." \
   bash setup-cloudflare.sh
   ```

   The script sets both required secrets, creates the KV namespace if needed, and
   deploys the Worker.

5. For real Google Play subscription verification, pass the optional secrets too:

   ```bash
   DATABASE_URL="postgresql://USER:PASSWORD@HOST/db?sslmode=require" \
   FIREBASE_WEB_API_KEY="AIzaSy..." \
   GOOGLE_PLAY_SERVICE_ACCOUNT_EMAIL="service-account@project.iam.gserviceaccount.com" \
   GOOGLE_PLAY_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n" \
   bash setup-cloudflare.sh
   ```

6. For manual internal subscription upserts, set `INTERNAL_API_KEY` manually:

   ```bash
   cd backend
   npx wrangler secret put INTERNAL_API_KEY
   ```

   You can also pass `INTERNAL_API_KEY="long-random-value"` together with the
   main deploy command in step 4.

## 4. Point Android to the Worker

Create or edit root `local.properties`:

```properties
sdk.dir=/path/to/android/sdk
syncBaseUrl=https://<name>.<cloudflare-subdomain>.workers.dev
```

Rules:

- `syncBaseUrl` must not end with `/`.
- Release builds require an explicit `syncBaseUrl`.
- Debug builds fall back to `https://devicedna-sync.workers.dev` only for local
  convenience.

Build:

```bash
./gradlew :android:assembleDebug
```

For release or CI:

```bash
./gradlew :android:bundleRelease -PsyncBaseUrl=https://<name>.<cloudflare-subdomain>.workers.dev
```

## 5. Configure Premium Subscriptions

Skip this section only if the handed-over product will not use real Google Play
subscriptions yet.

1. In Play Console, create subscription product:

   ```text
   devicedna_premium
   ```

2. If the product id is different, set it in both places:

   ```toml
   # backend/wrangler.toml
   GOOGLE_PLAY_PREMIUM_PRODUCT_ID = "your_product_id"
   ```

   ```properties
   # local.properties
   premiumSubProductId=your_product_id
   ```

3. Grant the backend service account access to the app in Play Console.

4. Set Cloudflare secrets:

   ```bash
   cd backend
   npx wrangler secret put GOOGLE_PLAY_SERVICE_ACCOUNT_EMAIL
   npx wrangler secret put GOOGLE_PLAY_PRIVATE_KEY
   ```

5. Build release with real billing enabled. Release defaults to real billing:

   ```bash
   ./gradlew :android:bundleRelease -PsyncBaseUrl=https://<worker-url>
   ```

For debug testing against the backend dev subscription endpoint:

```properties
devSubscriptionUseBackend=true
```

Then run local Worker with `DEV_SUBSCRIPTIONS_ENABLED="true"` in
`backend/.dev.vars`. Never enable `DEV_SUBSCRIPTIONS_ENABLED` in production.

### Optional: real-time cancellation/refund updates (RTDN)

This step is **optional**. Without it, a normal cancellation still works correctly —
premium lapses on its own at `expires_at`. Enable RTDN only if you want Google-side
**refunds/revocations** to drop premium almost immediately instead of at period end.

No database migration is required: the webhook reuses `user_subscriptions`
(`latest_purchase_token` already exists in `backend/drizzle/0002_*.sql`).

1. In Google Cloud (the project behind your Play service account) create a Pub/Sub
   topic, e.g. `play-rtdn`.

2. In Play Console -> Monetization setup -> Real-time developer notifications, set the
   full topic name and save. Google sends a one-off test notification to verify the link.

3. Set the webhook secret:

   ```bash
   cd backend
   npx wrangler secret put PLAY_RTDN_VERIFICATION_TOKEN
   ```

4. Create a Pub/Sub **push** subscription on that topic with the endpoint:

   ```text
   https://<worker-url>/play/notifications?token=<PLAY_RTDN_VERIFICATION_TOKEN>
   ```

   Pub/Sub push cannot send custom headers, so the secret travels in the query string.
   If `PLAY_RTDN_VERIFICATION_TOKEN` is unset, the endpoint returns `503` and changes
   nothing — leaving the app fully functional without RTDN.

## 6. Local Worker Development

Create `backend/.dev.vars` from the example:

```bash
cd backend
cp .dev.vars.example .dev.vars
```

Fill at least:

```properties
DATABASE_URL="postgresql://USER:PASSWORD@HOST/db?sslmode=require"
FIREBASE_WEB_API_KEY="AIzaSy..."
```

Then run:

```bash
npm run dev
```

`FIREBASE_PROJECT_ID` still comes from `backend/wrangler.toml`, so it must be set
there before local auth tests.

## 7. Verify End to End

1. Check Worker health:

   ```bash
   curl https://<worker-url>/
   ```

   Expected:

   ```json
   {"ok":true,"service":"devicedna-sync"}
   ```

2. Install and sign in:

   ```bash
   ./gradlew :android:installDebug
   adb logcat -s DeviceSync AuthRepository
   ```

   Expected after sign-in:

   ```text
   DeviceSync: syncIfNeeded(force=false)
   DeviceSync: Synced device <androidId> at <timestamp>
   ```

3. Watch Worker logs:

   ```bash
   cd backend
   npx wrangler tail
   ```

   Expected:

   ```text
   POST /auth/google 200
   GET /v1/me 200
   POST /v1/sync 200
   ```

4. Check Neon:

   ```bash
   cd backend
   DATABASE_URL="postgresql://USER:PASSWORD@HOST/db?sslmode=require" node -e "
   const { neon } = require('@neondatabase/serverless');
   (async () => {
     const sql = neon(process.env.DATABASE_URL);
     console.log(await sql\`select firebase_uid, email from users\`);
     console.log(await sql\`select user_uid, android_id, last_synced_at from devices\`);
   })();
   "
   ```

5. Relaunch the app.

   If the device snapshot did not change, logs should show `UpToDate` instead of
   pushing the full snapshot again.

## Common Failures

| Symptom | Most likely cause | Fix |
|---|---|---|
| `/auth/google` returns `auth_failed` | Wrong or missing `FIREBASE_WEB_API_KEY` | Set Cloudflare secret from the same Firebase project |
| `/v1/*` returns `invalid_token` | `FIREBASE_PROJECT_ID` does not match `google-services.json` | Update `backend/wrangler.toml` and redeploy |
| Worker deploy succeeds but auth fails | Old script/deploy did not set `FIREBASE_WEB_API_KEY` | Run current `backend/setup-cloudflare.sh` or `wrangler secret put FIREBASE_WEB_API_KEY` |
| Release build fails with missing `syncBaseUrl` | Backend URL was not configured | Add `syncBaseUrl` to `local.properties` or pass `-PsyncBaseUrl=...` |
| Premium verification returns `google_play_not_configured` | Google Play service account secrets are missing | Set `GOOGLE_PLAY_SERVICE_ACCOUNT_EMAIL` and `GOOGLE_PLAY_PRIVATE_KEY` |
| Dev subscription endpoint returns `dev_subscriptions_disabled` | `DEV_SUBSCRIPTIONS_ENABLED` is not set locally | Set it only in `backend/.dev.vars` for local testing |
| RTDN webhook returns `play_rtdn_not_configured` (503) | `PLAY_RTDN_VERIFICATION_TOKEN` secret is not set | Set it via `wrangler secret put` (RTDN is optional) |
| RTDN webhook returns `unauthorized` (401) | Pub/Sub push URL `?token=` does not match the secret | Re-create the push subscription with the correct token |

## Files To Clean Before Handover

Do not give the buyer your private configs or old backups.

Remove or regenerate:

- `android/google-services.json`
- `ios/DeviceDNAApp/GoogleService-Info.plist`
- any `*.bak` Firebase config files
- `backend/.dev.vars`
- local keystores
- real Neon connection strings
- old Cloudflare Worker secrets

The repository contains placeholders for handover:

- `.firebaserc` uses `YOUR_FIREBASE_PROJECT_ID`
- `backend/wrangler.toml` uses `YOUR_FIREBASE_PROJECT_ID`
- `backend/wrangler.toml` uses `REPLACE_AFTER_wrangler_kv_namespace_create`
- `local.properties.example` has no machine-specific paths

After handover, the new owner should rotate any credentials that were ever
visible to the old owner.
